## Context（背景）

当前仓库除了 README 之外还没有前端应用代码。本次需求是一个小型公开个人链接主页，主要用于展示个人信息和几个外部链接，因此静态 HTML/CSS 是最简单、最合适的实现方式。

## Goals / Non-Goals（目标 / 非目标）

**Goals（目标）：**
- 提供一个可直接用浏览器打开的响应式主页。
- 个人文案和链接直接写在 HTML 中，方便后续手动替换。
- 参考 Darshan Bhuva 个人站的极简开发者作品集视觉样式，不添加框架、包管理配置或构建步骤。
- 加入轻量自动化检查，确认页面包含必需的可见结构。

**Non-Goals（非目标）：**
- 不做后端、CMS、统计分析、登录认证或部署配置。
- 不添加 JavaScript 交互，除非后续出现明确需求。
- 不填入真实个人信息，只保留可替换的占位内容。

## Decisions（技术决策）

- 在仓库根目录使用 `index.html` 和 `styles.css`。
  - 理由：项目目前为空，需求也是静态页面，用两个文件即可满足。
  - 备选方案：React/Vite 应用。该方案会增加初始化和构建成本，但对当前需求没有明显收益。
- 使用语义化 HTML，包括 `main` 个人卡片、标题、简介段落和链接。
  - 理由：结构清晰，也更利于无障碍访问和自动化检查。
  - 备选方案：全部用 `div` 拼页面。这样语义较弱，不利于维护和辅助技术识别。
- 页面结构参考本地 `bio-link-page/src/index.html` 的组织方式。
  - 理由：该参考页更接近 Linktree 型个人主页，包含居中容器、资料区、头像环、纵向链接卡片、复制 toast 和 footer。
  - 取舍：迁移布局、交互组织和真实图标表达，但不迁移参考页的发光背景、扫描线或配色，避免破坏当前视觉风格。
- 页面尺寸进一步贴近本地参考页。
  - 理由：用户反馈排版和空间大小差异过大，因此容器宽度、body padding、头像环、卡片间距和 footer 间距应以参考页为基准。
  - 取舍：参考页使用外部 Font Awesome 和装饰光球；本项目只引入 Font Awesome CDN 用于真实图标，不添加离散光球装饰，保持页面轻量。
- 使用 CSS 变量、响应式尺寸和极简 landing 布局。
  - 理由：在保持简单的同时，让页面更接近开发者作品集首屏，而不是信息卡片面板。
- 使用黑色背景、浅蓝灰大标题、绿色强调、灰蓝正文和绿色描边按钮。
  - 理由：这些元素贴近参考站的极简开发者作品集质感，比 AIHOT 的资讯卡片风更适合个人链接主页。
  - 备选方案：继续保留 AIHOT 卡片资讯风。该方案更像 dashboard，不够像个人主页。
- 使用 Node 内置模块写一个小结构测试。
  - 理由：不引入测试框架，也能验证页面是否包含必需内容。
- 使用 `script.js` 承载页面交互。
  - 理由：微信复制、风格切换和访问次数都属于小型客户端行为，原生 JavaScript 足够。
  - 备选方案：引入前端框架。当前页面规模很小，框架会增加不必要复杂度。
- 使用 Font Awesome CDN 展示链接图标。
  - 理由：用户希望把文字缩写替换为真实图标，本地参考页也使用该方案；CDN 引入成本低，不需要构建步骤。
  - 取舍：离线或 CDN 失败时图标可能不可见，但链接文字仍然可读。
- 右上角按钮切换视觉风格，不再切换白色主题。
  - 理由：用户明确不要白色模式，而是切换到参考页风格。
  - 备选方案：保留深/浅色模式。该方案与最新反馈冲突。

## Risks / Trade-offs（风险 / 取舍）

- 当前姓名、简介和链接都是占位内容。-> 在 `index.html` 中保留明显占位值，方便后续替换。
- 微信没有统一的个人网页链接格式。-> 先放置 WeChat 按钮占位，后续可改成二维码页或其他联系方式链接。
- 视觉审核有一定主观性。-> 结合结构测试和浏览器截图检查，降低明显布局问题的风险。
- 参考站是完整作品集，当前页面是 Linktree。-> 只借鉴首屏排版、色彩、按钮和文字层级，不复制导航、项目区或长页面结构。
- 本地参考页带有额外装饰和外部图标库。-> 只引入 Font Awesome 显示真实图标，其他装饰仍保持当前黑底绿边设计。
- 剪贴板 API 在部分本地文件环境可能受限。-> 增加 textarea fallback，仍显示明确状态提示。
- 音乐按钮已从最新需求中移除。-> 删除 Web Audio 相关代码，避免无用交互和潜在浏览器限制。
